## Hello World

这是一篇测试文章，用于验证博客发布功能是否正常。发布成功后，您将能够在博客首页看到本篇文章的卡片，点击后可阅读完整内容。

### 背景与问题定义

在搭建个人博客系统后，首要任务就是验证发布流程是否可靠。本篇文章旨在模拟真实发布场景，从内容创建、渲染到部署，测试整个链路的连通性。

### 核心观点

通过发布一篇简单的“Hello World”文章，可以快速暴露基础配置与部署环节中的常见问题，例如：

- Frontmatter 格式错误
- 静态资源路径无效
- 部署脚本失败
- 缓存未清除

### 测试环境

本博客基于 [Astro](https://astro.build) 框架构建，主题为 Firefly，托管于 GitHub Pages 并通过 GitHub Actions 自动部署。

| 标签与分类 | 展示对应标签和分类 | ✅ |

### 测试步骤

1. 编写 Markdown 文章，添加正确 Frontmatter。
2. 将文章置于 `src/content/posts/` 目录。
3. 提交代码并推送至远程仓库。
4. 观察 GitHub Actions 运行状态。
5. 访问博客 URL 验证页面展示。

> 注意：如果部署后文章未显示，请检查 `draft` 字段是否为 `false`，以及构建日志中是否包含错误信息。

### 对比分析

手动部署与 CI/CD 自动部署各有优劣：

- **手动部署**：灵活性高，适合快速验证，但容易遗漏步骤。
- **CI/CD 自动部署**：一致性高，减少人为失误，但首次配置稍复杂。

对于本博客，推荐使用 GitHub Actions 自动部署，确保每次推送都能同步更新。

### 实践建议

- 首次发布前，先本地运行 `npm run build` 检查构建是否成功。
- 使用 `draft: true` 保存草稿，待确认无误后再改为 `false`。
- 部署后清除浏览器缓存或使用隐私窗口查看。

### 风险与局限

本测试仅覆盖基础文章发布链路，未包含以下场景：

- 文章密码保护
- 自定义 slug 与国际化
- 复杂 Markdown 扩展（Mermaid、KaTeX）
- 评论系统后端异常

建议在使用高级功能时，分别编写对应测试文章。
